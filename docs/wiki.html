I will use the word xlet throughout this wiki to refer to any Cinnamon applet, extension or desklet.

GitHub tips & tricks

A series of how-tos about advanced (from a beginners standpoint) GitHub tasks.

Contribute to someone's repository - Source

Say you want to contribute changes to someone else’s repository (eg, Cinnamon's repository). The basic concept is as follows:

  • Fork the original repository into your GitHub account.
  • Clone your fork locally and make changes to it.
  • Commit and push the local changes to your on-line repository.
  • Create a pull request to send to the original repository.
  • Pull requests can then be reviewed and approved/rejected by the repository's administrators.

Detailed steps

  1. Go to the repository on GitHub and click the Fork button at the top right. You’ll now have your own copy of that repository in your GitHub account.

  2. Open a terminal/shell inside a directory where you want to store your cloned repositories and type:

     git clone git@github.com:your_github_username/Cinnamon

  3. You’ll now have a local copy of your version of Cinnamon's repository. Change into that project directory (Cinnamon):

     cd Cinnamon

  4. Add a connection to the original owner’s repository.

     git remote add upstream git://github.com/linuxmint/Cinnamon

    Note: The remote's name can be anything other than upstream, but I will use only this name. Mainly because it makes sense. And also because I will use it in several parts of this Wiki.

  5. At this point, we can start making changes to files to fix bugs and/or add features and then commit and push those changes to our fork of the repository. To keep things organized on both ends (our fork and the original repository), it's better to work with branches and keep the master branch of your fork untouched.

    1. Make sure that the currently checked out branch is master.

       git checkout master

    2. Create a new branch and check it out.

       git checkout -b my-branch-with-a-fix

    3. Make all changes, commit and push (always create detailed commit messages). Then proceed with point 6 bellow.

  6. Go to your version of the repository on GitHub.

    1. You most likely will see your recently pushed commit/branch and next to it a button called Compare & pull request. Click that button.
    2. You will be presented with a comparison between the original repository and your fork and with a text box that will let you set a title and a message for your pull request.
    3. After setting a succinct and informative title and a detailed explanation of the changes introduced by your pull request, click the Create pull request button.

  7. At this point your pull request is queued to be reviewed and approved/rejected by the repository's administrators.

Keep your forked repositories synchronized with the original repositories

Perform these steps periodically to ensure that you are working with updated code.

  1. Make sure you are on the branch you are supposed to be, if you don’t know, you want master. To ensure you’re on the local master branch, do this:

     git checkout master

  2. Fetch updates from the original repository into your local copy of your fork.

     git fetch upstream

  3. Re-base the local master branch from the upstream's master branch.

     git rebase upstream/master

  4. Synchronize the local repository with the on-line one.

     git push origin master

Avoid user/password prompts when pushing - Source

1. Generate an SSH key

Linux/Mac

Open terminal to create ssh keys:

cd ~                 # Your home directory
ssh-keygen -t rsa    # Press enter for all values

For Windows

(Only works if the commit program is capable of using certificates/private & public ssh keys)

  • Use Putty Gen to generate a key
  • Export the key as an open SSH key

Here is a walkthrough on putty gen for the above steps

2. Associate the SSH key with the remote repository

This step varies, depending on how your remote is set up.

  • If it is a GitHub repository and you have administrative privileges, go to settings and click 'add SSH key'. Copy the contents of your ~/.ssh/id_rsa.pub into the field labeled 'Key'.

  • If your repository is administered by somebody else, give the administrator your id_rsa.pub.

3. Set your remote URL to a form that supports SSH 1

If you have done the steps above and are still getting the password prompt, make sure your repo URL is in the form...

git+ssh://git@github.com/user_name/repo_name.git

...as opposed to...

https://github.com/user_name/repo_name.git

To see your repository's current URL, run:

git remote show origin

You can change the URL with:

git remote set-url origin git+ssh://git@github.com/user_name/repo_name.git

Summarized steps for Linux

1. Create the SSH keys

ssh-keygen -t rsa -f ~/.ssh/file_name -b 4096 -C "Comment"

It will ask for passphrase and then create the files ~/.ssh/file_name and ~/.ssh/file_name.pub.

2. Add the public key ~/.ssh/file_name.pub to GitHub account SSH section.

3. Add the private key to ssh-agent

ssh-add ~/.ssh/file_name

4. Add remote url to your local git repository

git remote set-url origin git+ssh://git@github.com/user_name/repo_name.git

Rewriting history - Source

Reword the last commit message

One of the simplest history rewrites we can do with git is changing the last commit message. Let’s say right after making a commit you find a typo in its description, or you find a better way of describing the changeset. To make the correction you run:

git commit --amend

It will open an editor with the last commit message, so you can modify it. After saving, a new commit will be created with the same changes and the new message, replacing the commit with the previous message.

This can be useful to include files you forgot to track, or include modifications to the files you just commited. To do so, you can add the changes and then perform the amend:

git add file1.md folder/file2.conf
git rm file3.txt
git commit --amend

Aside from editing the commit message, the new commit will contain the changes specified with git add and git rm.

You can now change the last commit of your repository to include newer changes to the files, and/or to improve the commit message. But don’t start amending all-the-things before understanding the last section titled DANGER.

Rebase and interactive rebase

To reword other commit messages, squash commits together and rebase on top of master, git rebase is used. git rebase re-applies commits, one by one, in order, from your current branch onto another. It accepts several options and parameters, so that’s a tip of the iceberg explanation, enough to bridge the gap in between StackOverflow or GitHub comments and the git man pages.

An interesting option it accepts is --interactive (-i for short), which will open an editor with a list of the commits which are about to be changed. This list accepts commands, allowing the user to edit the list before initiating the rebase action.

Reword other commit messages

Let’s say I want to reword any of the last 4 commits of a repository. I then run git rebase -i HEAD~4, and here is what I see:

pick 07c5abd Added new feature 1.
pick 3e7ee36 Fixed an error.
pick de9b1eb Fixed a type.
pick fa20af3 Added new feature 2.

# Rebase 8db7e8b..fa20af3 onto 8db7e8b
#
# Commands:
#  p, pick = use commit
#  r, reword = use commit, but edit the commit message
#  e, edit = use commit, but stop for amending
#  s, squash = use commit, but meld into previous commit
#  f, fixup = like "squash", but discard this commit's log message
#  x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#
# If you remove a line here THAT COMMIT WILL BE LOST.
#
# However, if you remove everything, the rebase will be aborted.
#
# Note that empty commits are commented out

We see the four last commits, from older to newer. See the comment below the list of commits? Good job explaining, git! pick (p for short) is the default action. In this case it would reapply the commit as is, no changes in its contents or message. Saving (and executing) this file would make no changes to the repository.

If I say reword (r for short) in a commit I want to edit:

pick 07c5abd Added new feature 1.
pick 3e7ee36 Fixed an error.
r de9b1eb Fixed a type.
pick fa20af3 Added new feature 2.

When I save and quit the editor, git will follow the described commands, landing myself into the editor again, as if I had amended commit de9b1eb. I edit that commit message (note that it says type instead of typo), save and quit the editor, and here is the output:

hostname branch-name % git rebase -i HEAD~4
[detached HEAD dd62a66] Fixed a typo.
 Author: No Name
 Date: Fri Oct 31 10:52:26 2017 -0500
 2 files changed, 39 insertions(+), 42 deletions(-)
Successfully rebased and updated refs/heads/branch-name.

Now the commit message says "Fixed a typo." instead of "Fixed a type.".

You can now change the message of any commit you want. You may do so, just make sure you understand the DANGER section.

Squash commits together

Two other commands rebase interactive offers us are:

  • squash (s for short), which melds the commit into the previous one (the one in the line before).
  • fixup (f for short), which acts like “squash”, but discards this commit’s message.

We’ll continue to work on the rebase example we worked before. We had four commits, ther commits that add new features and one commit that fixes a typo on the first commit:

pick 07c5abd Added new feature 1.
pick 3e7ee36 Fixed an error.
pick de9b1eb Fixed a typo.
pick fa20af3 Added new feature 2.

Let’s say I want to meld the first three commits together, because they belong to the same logical changeset. We’ll want to keep the first commit message, and squash the two subsequent commits into the previous one. I change pick to squash where appropriate:

pick 07c5abd Added new feature 1.
s 3e7ee36 Fixed an error.
s de9b1eb Fixed a typo.
pick fa20af3 Added new feature 2.

Save, and I land into the editor to decide the commit message of the melded three commits (see how they are concatenated one after the other):

# This is a combination of 3 commits.
# The first commit's message is:
Added new feature 1.

# This is the 2nd commit message:

Fixed an error.

# This is the 3rd commit message:

Fixed a typo.

# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
#
# Author:    No Name
# Date:      Tue Sep 2 09:39:07 2017 -0500
#
# rebase in progress; onto 71d4789
# You are currently editing a commit while rebasing branch 'branch-name' on '71d4789'.

I decide to remove the third commit message, and add a more relevant note to the second commit message. Save the editor, and the four commits were transformed into two: the one with the commit message "Added new feature 1" and the other one with commit message "Added new feature 2".

We could have used the fixup command, if we had seen earlier that we want the changes, but not the commit message, of the third commit. In that case, the commands would have looked like:

pick 07c5abd Added new feature 1.
s 3e7ee36 Fixed an error.
f de9b1eb Fixed a typo.
pick fa20af3 Added new feature 2.

When saved, the editor would have included the third commit message already commented out for us:

# This is a combination of 3 commits.
# The first commit's message is:
Added new feature 1.

# This is the 2nd commit message:

Fixed an error.

# This is the 3rd commit message:

# Fixed a typo.

# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
#
# Author:    No Name
# Date:      Tue Sep 2 09:39:07 2017 -0500
#
# rebase in progress; onto 71d4789
# You are currently editing a commit while rebasing branch 'branch-name' on '71d4789'.

Save, and outputs:

[detached HEAD 809241b] Added new feature 1.
 Author: No Name
 Date: Tue Sep 2 09:39:07 2017 -0500
 2 files changed, 1429 insertions(+), 1 deletion(-)
Successfully rebased and updated refs/heads/branch-name.

Result is the same: 2 commits instead of 4, each with a single, feature added.

You can now merge commits together. As always, be mindful of the DANGER section.

Rebase on top of master

We fork a repository, start working on a feature branch, and master in the upstream project moves ahead. Our history looks like:

       A---B---C feature
     /
D---E---F---G upstream/master

The original repository maintainer asks as to rebase on top of master, so we fix any merge conflicts that may arise between both branches, and keep our changeset together. The maintainer would like to see a history like:

               A---B---C feature
             /
D---E---F---G upstream/master

We want to reapply our commits, one by one, in order, onto upstream’s master. Sounds like the description of the rebase command! Let’s see what commands would land us into the desired scenario:

It is assumed that you already have your forked repository set up to work with an upstream remote. See Contribute to someone's repository

# Fetch latest commits from `upstream` (the original fork)
git fetch upstream

# Checkout our feature branch
git checkout feature

# Reapply it onto upstream's master
git rebase upstream/master

# Fix conflicts, then `git rebase --continue`, repeat until done
# Push to our fork
git push --force origin feature

Your feature branch will be applied on top of latest master of the original fork.

DANGER: You are rewriting history

See the --force in the last git push command? That means we are overwriting repository’s history. This is always safe to do in commits we don’t share with other team members, or in branches that belong to us.

But if you force push editions that were already shared with the team (commits that exist outside of my repository), then everyone’s branch gets out of sync.

Rewriting history means abandoning existing commits and creating new ones, that may be very similar but are different. If others base work on your previous commits, and then you rewrite and force-push your commits, your team members will have to re-merge their work (if they notice the potential loss).

So rewrite git history, provided rewritten commits exist only in your repository, or you and your team know that no one else should base work off of them.

You now know how to rebase while being a good citizen.

Working with big repositories - Source

Note: The minimum git version required for this to work is 1.9.

You can combine the sparse checkout and the shallow clone features. The shallow clone cuts off the history and the sparse checkout only pulls the files matching your patterns.

  1. Create an empty local repository.

     git init <repository>
 cd <repository>

  2. Add remote URL to it.

     git remote add origin https://github.com/user_name/repository.git

  3. Enable sparce checkout.

     git config core.sparsecheckout true

  4. Specify the directories and/or files that one wants to checkout.

     echo "directory1/*" >> .git/info/sparse-checkout
 echo "directory2/*" >> .git/info/sparse-checkout
    • One could simply edit the .git/info/sparse-checkout file with any text editor and add the list of files/directories to it.

  5. Create a shallow clone.

     git pull --depth=1 origin master

At this point, all the repository data will be downloaded, but only the files/folders specified inside the .git/info/sparse-checkout file will be checked out.

Closing issues via commit messages - Source

You can include keywords in your commit messages to automatically close issues in GitHub.

Closing an issue in the same repository

To close an issue in the same repository, use one of the keywords in the list below followed by a reference to the issue number in the commit message. For example, a commit message with Fixes #45 will close issue 45 in that repository once the commit is merged into the default branch.

If the commit is in a non-default branch, the issue will remain open and the issue will be referenced with a tooltip.

Keywords for closing issues

The following keywords will close an issue via commit message:

  • close
  • closes
  • closed
  • fix
  • fixes
  • fixed
  • resolve
  • resolves
  • resolved

Closing an issue in a different repository

To close an issue in another repository, use the user_name/repo_name#issue_number syntax, as described in "Autolinked references and URLs".

For example, including Closes example_user/example_repo#76 will close the referenced issue in that repository, provided you have push access to that repository.

Closing multiple issues

To close multiple issues, preface each issue reference with one of the above keywords.

For example, This closes #34, closes #23, and closes example_user/example_repo#42 would close issues #34 and #23 in the same repository, and issue #42 in the "example_user/example_repo" repository.

Xlet instalation

How to install xlets in Cinnamon

Note: I will use the word xlet throughout this wiki to refer to any Cinnamon applet, extension or desklet.

Some generalities

  • Xlets in Cinnamon can be easily installed from Cinnamon Settings. These xlets are hosted on the Spices website.
  • By default, all xlets are installed into the user's home folder inside ~/.local/share/cinnamon/[xlet]. To install them globally for all users, refer to the section called Installing xlets for all users.
  • The name of an xlet folder is also its UUID (Universally Unique IDentifier). So, without exceptions, an xlet folder always have to be named exactly as it is declared inside its metadata.json file. NEVER RENAME AN XLET FOLDER!!!
  • Never delete any folder or file of any type inside an xlet folder unless you are absolutely sure that that folder/file isn't used by the xlet itself.
  • Xlets can be downloaded from the Spices website and installed manually on their respective locations.

Applets locations

  • User specific location: ~/.local/share/cinnamon/applets
  • System wide: /usr/share/cinnamon/applets

Extensions locations

  • User specific location: ~/.local/share/cinnamon/extensions
  • System wide: /usr/share/cinnamon/extensions

Desklets locations

  • User specific location: ~/.local/share/cinnamon/desklets
  • System wide: /usr/share/cinnamon/desklets

Installing xlets for all users

  • For xlets to be available for all users, they can be installed into /usr/share/cinnamon/[xlet].
  • For all users be able to see the xlets in their Cinnamon Settings, the xlet folder has to have the right permissions.
cd /usr/share/cinnamon/[xlet]
sudo chmod ugo+rx xlet@folder_name
Important note: This is only worth doing it for xlets not published on Spices website. Because every time an update is found by the xlets managers, it will be installed on the user specific location, overriding the xlet installed on the system wide location.

Xlet localization

How to localize xlets

Note: I will use the word xlet throughout this wiki to refer to any Cinnamon applet, extension or desklet.

For end users/contributors

Some technical facts

These facts are based on documentations and source code that I read.

  • .po files can be added to a "po" folder in your xlet's directory, and will be compiled and installed into the system when the xlet is installed via Cinnamon Settings.
  • If the xlet is installed manually and not trough Cinnamon Settings, localizations can be installed with the command cinnamon-json-makepot -i executed from a terminal opened inside the xlet's folder. Assuming that a folder called po with .po files exists inside the xlet's folder.
  • Most xlets with support for localizations comes with a template file (a .pot file) that can be used to create .po files. But, in case that a .pot file isn't available, an existent .po file can be copied and renamed and then make the translations on that newly created file.

How to create localization files (.po files)

The easiest (and safer) way would be using Poedit. This program can be installed from the repositories of almost all Linux distributions and also can be installed on Windows and OS X.

Note: .po and .pot files are simply text files. They could be edited manually with a text editor, but I will only explain .po and .pot files manipulation using Poedit. Editing .po files manually could lead to errors that would render the .po file useless. And a .pot file should never be edited by other than the xlet's developer.

Poedit usage

Creating a translation file from scratch
  • Open Poedit and click on Create new translation.
    • Open a .pot file if it was made available for the xlet. This file is created by the xlet developer from the source code of the xlet. Therefore, this file will be the most up-to-date and will contain the most amount of strings to translate from.
    • If no .pot file if available, any .po file containing any locale can be used to create a new translation (.po file).
  • Once the file is loaded, Poedit will ask you to choose a language for the translation.
  • At this point, the .po file can be saved and the name will automatically reflect the locale code.
  • Poedit usage is pretty much straight forward:
    • There is the Source text box that contains the string to be translated (usually in English).
    • There is the Translation box that will contain the translated text.
    • There is a button called Add comment that will allow you to add comments for other translators to read. Never delete comments!!!
    • If there is a .mo file saved when you save your .po file, the .mo file can be deleted (it isn't used by Cinnamon xlets). You could also disable MO files creation from Poedit preferences (go to Edit > Preferences > General tab and uncheck the option called Automatically compile MO file when saving).

Updating and existing translation file
  • Open the .po file with Poedit.
  • Go to the menu Catalog and choose Update from POT file... menu item.
  • Select a .pot file from which your .po file was based on.
  • You will be presented with a window with 2 lists (New strings and Obsolete strings).
    • New strings are strings that will be added to the .po file and will need to be translated.
    • Obsolete strings are strings that aren't used any more and will be removed from the to the .po file.

Help files localization

All the xlets in the CinnamonTools repository are shipped with a help file (HELP.html file found in the root directory of each xlet). I recently added the ability to localize these help files. So, any translator that is willing to translate any of these xlets, will also find strings on the xlets .pot files that are used to generate these HELP.html files.

Basic technicalities about the help files

  • The strings used to generate the HELP.html files are used to build on the fly Markdown markup. And then, that Markdown markup is used to generate the final HTML document.
  • Translators might find several strings commented as follows:
    • # TO TRANSLATORS: MARKDOWN string. Respect formatting.
  • I use Markdown markup as opposed to pure HTML tags for one simple reason. Translators will not have to worry about the proper use of HTML tags. If a translator would use wrongly an HTML tag (not properly closed, for example), the generated HELP.html file will probably completely break. If a translator would wrongly use Markdown markup, the worse that could happen is that the characters used to mark up text will be shown, without breaking the HELP.html document rendering.
  • I will limit the exposing of Markdown markup on the translatable strings as much as possible. Basically, what the translator needs to know about Markdown markup is the following:
    • **This is bold text** and is rendered like this: This is bold text
    • *This is italic text* and is rendered like this: This is italic text
    • [This is a link text](http://link.url) and is rendered like this: This is a link text
    • These are [[Keyboard]] [[keys]] and are rendered like this: Keyboard keys

Testing your translations of the help files

Any translator that wants to test their translations of the strings used by the HELP.html files can do so by generating the HELP.html file using the newly created/updated .po files as follows.

  • Locate the file called create_localized_help.py. This script is located two levels up from the actual xlet folder.
  • Open a terminal in that location and execute the script as follows: create_localized_help.py --dev.
  • This script should only be executed from the location it is found and from a fully checked out branch of the CinnamonTools repository.
  • After executing the script, a file called XLET_UUID-HELP.html will be created based on all .po files found inside the po folder of an xlet.
  • The XLET_UUID-HELP.html file will be located inside a temporary folder in the root of the repository folder (/CinnamonTools/tmp/help_files).
  • Translators shouldn't update the final HELP.html file, so they shouldn't add this file to their pull requests containing new/updated translations. This is to avoid to force the translators to rebase their pull requests every time more than one translator updates the HELP.html file of the same xlet.
  • The folder called tmp in the root of the CinnamonTools repository is automatically created and used by several scripts inside this reposiory. It is already listed in the .gitignore file, so you don't need to worry about any files inside that folder to show up on your commits.
